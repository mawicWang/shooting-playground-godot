# 🛰️ 🔫 Shooting Playground v1.0: 逻辑涌现 (Logic Emergence) 策划方案

本文档详细描述了项目的下一个重大版本方向。该版本将游戏从传统的"静态塔防"转变为基于**子弹驱动**和**模块化编程**的动态逻辑系统。

---

## 1. 核心哲学：从"炮塔防御"到"逻辑布线"

在 v1.0 中，防御过程被抽象为**信号处理**：
*   **子弹 = 信号数据包**：不再只是伤害数值，而是携带"能量、频率、属性"的载体。
*   **炮塔 = 逻辑算子**：玩家在 5x5 的空间内不是在"造塔"，而是在"编程"。通过子弹在塔群间的吸收与再射出，形成一个能够自我增强的防御算法。

---

## 2. 炮塔体系

### 2.1 数据驱动架构

每种炮塔由一个 `TowerData` Resource 定义。代码层面只有**一个通用 Tower 类**，不同塔的行为差异全部由数据决定。新增塔 = 新建一个 `.tres` 文件，不需要新脚本。

**TowerData 按需扩展**：只在实际需要时添加新字段，不为未来可能的功能预留空接口。

### 2.2 Stat Modifier 系统

炮塔和子弹的所有数值属性（攻速、伤害、速度等）统一使用 **"基础值 + 修饰器"** 机制：

*   `StatAttribute`：封装一个数值，持有 base_value 和 modifier 列表
*   `StatModifier`：一个修饰器，含 value、type（加法/乘法）、source（谁加的）
*   计算公式：`最终值 = (base + Σ加法修饰) × Π乘法修饰`
*   卸载模块/失去遗物时，按 source 移除对应 modifier，数值自动回退，无残留

这是模块系统和遗物系统的**共同基础设施**，必须先于它们建立。

### 2.3 玩家视角的塔类型（策划概念，非代码分类）

从玩家体验的角度，塔可能表现为不同的"角色"，例如：
*   **发射型**：系统的时钟信号，定时产生原始子弹。
*   **中继型**：被动触发。仅在被子弹击中时，按照设定的逻辑重新发射。
*   **终端型**：吸收子弹，触发特殊效果（引力场、全场增益、资源转化等）。

这些不是代码中的类型枚举，而是不同 TowerData 配置组合涌现出的游戏体验。

---

## 3. 逻辑模块 (Modules) —— 塔的"固件"

这些是插在塔上的功能插件，决定了"编程感"的深度。

> **关键实现细节**：Module 是 Resource，安装到塔时必须 `duplicate()` 隔离实例状态。
> 例如 DividerModule 需要维护命中计数器，多个塔装同一种模块时各自独立计数。

### 3.1 计算类
*   **乘法器 (Multiplier)**：新子弹能量 = 原子弹能量 × 1.2（或更高倍率）。
*   **加速器 (Overclocker)**：显著提升射出子弹的速度。

### 3.2 逻辑类
*   **分频器 (Divider)**：计数器机制。例如：每被击中 3 次，才射出一颗超强子弹。
*   **过滤器 (Filter)**：属性判断。例如：只有特定属性（如火属性）的信号能通过，用于构建多路径分支逻辑。

---

## 4. 遗物系统 (Relics) —— 机制级全局 Buff

类似《杀戮尖塔》的遗物机制，侧重于修改规则而非纯数值，例如：

*   **"量子纠缠"**：场上所有同类型的终端机（0炮口塔）共享能量进度，一人充能，全场受益。
*   **"超导回路"**：子弹在塔之间传递时，飞行速度不衰减，且每次转发都会释放微小静电电击周围敌人。
*   **"递归逻辑"**：如果子弹在塔群中形成闭环路径（A→B→C→A），每循环一圈，子弹的体积和威力增大 10%。
*   **"多线程"**：所有的"发射源"底座在主炮口的正后方额外增加一个辅助炮口。

---

## 5. 游戏循环 (The Roguelike Loop)

1.  **情报预警**：提示下一波敌人入侵的方向（左/右/上/下）及单位类型。
2.  **部署 (Deployment)**：
    *   玩家根据库存中的炮塔，在网格上部署，炮塔的设计线路变成一种隐藏式的布线。
    *   当玩家获得模块时，可以安装到炮塔上，组合出各种各样的效果，模块随时可以取下。
3.  **运行/作战 (Execution)**：点击开始，观察防御逻辑是否按预期工作。
4.  **重构奖励 (Reward)**：三选一，例如
    *   [获得一种新的炮塔]
    *   [获得一个高级逻辑模块]
    *   [获得一个全局机制遗物]

---

## 6. 技术实现要点 (Roadmap)

*   [ ] **TowerData Resource + 数据驱动**：重构 tower.gd 为通用类，由 TowerData 配置行为。
*   [ ] **Stat Modifier 系统**：StatAttribute + StatModifier，所有数值属性统一管理。
*   [ ] **信号传递闭环**：重构 `bullet.gd` 携带 Data 字典，支持 `source_tower` 和 `last_sender` 区分。
*   [ ] **插槽系统**：实现 Resource 驱动的模块化逻辑，安装时 `duplicate()` 隔离状态。
*   [ ] **碰撞层扩展**：塔的信号接收区域（新碰撞层），让子弹能命中塔触发逻辑。
*   [ ] **实时旋转控制**：优化手机/Web 端的点击响应。
*   [ ] **视觉辅助**：添加"逻辑流向"淡入预览线。
*   [ ] **对象池系统 (Object Pool)**：复用子弹实例，保障中后期帧率稳定。
